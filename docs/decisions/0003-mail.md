# Which Email Service should we use for Transactional Email?

---

**Last Update:** 2026-05-31

---

## Context and Problem Statement

Collaborative features may require transactional email for use cases such as custom notifications. Since we are using Clerk for authentication, core flows like team invitations, account verification, and password reset are handled by Clerk natively. The question is therefore whether we need a separate email service at all, and if so, which one.

## Considered Options

* Clerk's built-in email (no separate service)
* Resend (standalone or as Clerk's custom email provider)
* Nodemailer + SMTP (self-hosted or provider relay)
* SendGrid

## Decision Outcome

Chosen option: **"Clerk's built-in email for now, with Resend deferred"**, because

* Clerk handles all auth-related emails (invitations, verification, password reset) out of the box — no additional service is needed for these flows.
* The only reason to add Resend would be application-level notifications (e.g. "a team member saved changes") or custom-branded auth emails, neither of which is required for an initial prototype.
* If custom branding or app notifications become necessary, Resend is the preferred service to add: it has a TypeScript-first API, a generous free tier, and can be plugged into Clerk as a custom SMTP provider.

## Pros and Cons of the Options

### Clerk's Built-in Email (no separate service)

#### Pros

* **Zero setup**: All auth emails work immediately with no configuration.
* **No additional service**: Reduces the number of external dependencies and infrastructure to manage.
* **Sufficient for a prototype**: Covers all required email flows for the initial collaborative feature set.

#### Cons

* **Clerk branding**: Emails are sent from a Clerk domain with Clerk's default styling unless a custom provider is configured.
* **No app-level notifications**: Cannot send custom application emails (e.g. session activity notifications) without adding a separate service.
* **Template limitations**: Email templates can only be edited within Clerk's dashboard.

### Resend (as Clerk's custom email provider)

#### Pros

* **Custom domain and branding**: Emails are sent from your own domain, removing Clerk branding.
* **TypeScript-first API**: Clean, fully typed SDK that fits naturally into the existing codebase.
* **Generous free tier**: 3,000 emails/month at no cost.
* **Future-proof**: The same Resend account can also be used for application-level notifications if needed.

#### Cons

* **Custom domain required**: Clerk only allows custom email providers when sending from a verified domain; a domain must be owned and configured in both Resend and Clerk.
* **Template control is still limited**: Even with Resend as the provider, email templates are edited in Clerk's dashboard — the full React Email workflow is not available for Clerk-triggered emails.
* **Setup overhead**: DKIM/SPF records must be configured and Clerk's SMTP settings updated; only worthwhile once branding matters.

### Nodemailer + SMTP

#### Pros

* **No external dependency**: Emails can be sent via any SMTP server.
* **Full control**: No third-party service involved in the delivery path.

#### Cons

* **Deliverability burden**: Managing SPF, DKIM, and DMARC records manually requires significant effort.
* **Dated API**: Less ergonomic and type-safe than modern alternatives.
* **Not compatible with Clerk**: Cannot be used to replace Clerk's email provider.

### SendGrid

#### Pros

* **Established and reliable**: Strong track record for high-volume delivery.
* **Feature-rich**: Analytics, template management, and marketing email features included.

#### Cons

* **Complexity**: Dashboard and API are more involved than necessary for simple transactional email.
* **Restrictive free tier**: Limits have historically been reduced over time.
* **Overhead**: Many features are irrelevant to this use case.
