# Which Email Service should we use for Transactional Email?

---

**Last Update:** 2026-05-31

---

## Context and Problem Statement

Collaborative features require transactional email for use cases such as team invitations, account verification, and password reset flows. We need an email delivery service that is easy to integrate, reliable, and suitable for a low-to-moderate volume application.

## Considered Options

* Resend
* Nodemailer + SMTP (self-hosted or provider relay)
* SendGrid

## Decision Outcome

Chosen option: **"Resend"**, because

* It provides a clean, TypeScript-first API that integrates well with our existing stack.
* The free tier (3,000 emails/month) is sufficient for the expected volume of this prototype.
* It supports sending from a custom domain with simple DNS setup and handles deliverability concerns (SPF, DKIM) automatically.
* React Email can be used alongside it to build and preview templates with full type safety, though plain HTML templates are also supported.

## Pros and Cons of the Options

### Resend

#### Pros

* **TypeScript-first API**: The SDK is fully typed, which fits naturally into our codebase.
* **Simple setup**: Domain verification and DNS configuration are straightforward, with deliverability handled by the service.
* **React Email integration**: Optional but powerful — email templates can be built as React components and previewed in the browser.
* **Generous free tier**: 3,000 emails/month and 100/day at no cost, which is sufficient for development and early usage.
* **Modern DX**: Clean API design with good documentation.

#### Cons

* **Vendor dependency**: Email delivery relies on Resend's infrastructure; outages or pricing changes are outside our control.
* **Newer service**: Less proven at very high volume compared to established providers like SendGrid or Postmark.

### Nodemailer + SMTP

#### Pros

* **No external dependency**: Emails can be sent via any SMTP server, including self-hosted solutions like Postfix or provider relays like Gmail.
* **Full control**: No third-party service involved in the email delivery path.

#### Cons

* **Deliverability burden**: Managing SPF, DKIM, DMARC records and avoiding spam filters requires significant additional effort.
* **No managed infrastructure**: A self-hosted SMTP server requires maintenance and monitoring.
* **Dated API**: Nodemailer's API is less ergonomic and type-safe compared to modern alternatives.

### SendGrid

#### Pros

* **Established and reliable**: Widely used with a strong track record for high-volume delivery.
* **Feature-rich**: Includes analytics, template management, and marketing email features.

#### Cons

* **Complexity**: The API and dashboard are more complex than necessary for simple transactional email.
* **Restrictive free tier**: The free tier has tighter limits and has historically been reduced over time.
* **Overhead**: Many features are irrelevant to our use case, adding unnecessary surface area.
