# Which Authentication Solution should we use?

---

**Last Update:** 2026-05-31

---

## Context and Problem Statement

Adding collaborative features requires user accounts and session management. Users need to be able to sign up, log in, and be associated with teams. We need an authentication solution that integrates well with our Nuxt 4 application and supports common OAuth providers, email/password login, and team/organisation management.

## Considered Options

* Clerk
* Auth0
* `@sidebase/nuxt-auth`
* `nuxt-auth-utils`

## Decision Outcome

Chosen option: **"Clerk"**, because

* Organizations are a first-class primitive in Clerk — team creation, invitations, membership roles, and a team switcher UI are included out of the box, mapping directly onto the collaborative features we want.
* Sign up, sign in, and account management UI components are pre-built, eliminating the need to build and style auth forms.
* The free tier (10,000 MAU) is sufficient for a prototype of this scale, and the time saved on auth implementation can be redirected to the collaborative editing features.
* User and team membership can be referenced in our Postgres schema via a `clerkOrgId`, meaning we do not need to model users or memberships ourselves.
* The Nuxt SDK is well-maintained and straightforward to integrate.

## Pros and Cons of the Options

### Clerk

#### Pros

* **Organizations as a first-class feature**: Team creation, invitations, role management, and a team switcher are included without any custom backend work.
* **Pre-built UI components**: Sign in, sign up, user profile, and org management screens are ready to use, saving significant frontend implementation time.
* **Minimal backend data modelling**: User and membership management is handled by Clerk; Postgres only needs to store a `clerkOrgId` reference.
* **Nuxt SDK**: Official SDK with composables and server-side helpers that integrate naturally with Nuxt.
* **Reduced complexity**: Email verification, password reset, OAuth, MFA, and session refresh are all handled by the service.

#### Cons

* **Vendor lock-in**: User data and session management are controlled by Clerk; migrating away would require exporting users and rebuilding the auth layer.
* **Recurring cost**: Free tier caps at 10,000 MAU; exceeding this requires a paid plan.
* **Less control**: Customising auth flows or data models beyond what Clerk exposes requires workarounds.

### Auth0

#### Pros

* **Mature and battle-tested**: Long track record in production across large organisations.
* **Broad protocol support**: OIDC, SAML, and enterprise SSO are well supported.

#### Cons

* **Organizations are a paid add-on**: The team/org management feature we want is not available on the free tier.
* **Complex dashboard**: Auth0's configuration surface is significantly more involved than Clerk's, adding setup overhead.
* **Dated DX**: SDK and documentation feel less modern compared to Clerk.

### `@sidebase/nuxt-auth`

#### Pros

* **Self-hosted**: No external service dependency; full control over user data.
* **Auth.js foundation**: Large ecosystem of OAuth providers and adapters.
* **Database sessions**: Sessions stored in PostgreSQL via a Prisma adapter.

#### Cons

* **No team primitives**: Organisation and membership management must be built from scratch in the database and API layer.
* **Auth.js complexity**: Notoriously difficult to configure correctly, particularly the Prisma adapter, callbacks, and JWT customisation.
* **UI must be built**: Sign in and sign up forms, email verification flows, and password reset all require custom implementation.
* **Nuxt 4 compatibility**: As a wrapper around a Next.js-oriented library, there can be lag or edge cases on major Nuxt releases.

### `nuxt-auth-utils`

#### Pros

* **Maintained by the Nuxt team**: First-party support ensures compatibility with the latest Nuxt versions.
* **Lightweight**: Minimal abstraction with secure session handling via `useUserSession()`.

#### Cons

* **No team primitives**: Same gap as `@sidebase/nuxt-auth` — all org and membership logic must be built manually.
* **OAuth only**: No built-in credentials support; email/password requires additional custom implementation.
* **Least feature-complete**: Smallest surface area of all options considered.
